# Архитектура почтового бота-автоответчика

Полное описание проекта: как устроен, как работает, что в каждом файле.

---

## 1. Что делает бот (в двух словах)

Бот следит за почтовым ящиком Outlook. Когда приходит письмо-заявка
«прошу направить реквизиты для оплаты курса», он:

1. распознаёт заявку и вытаскивает данные (курс, дата, организация, ИНН, КПП,
   адрес, ФИО слушателей);
2. ищет курс(ы) в прайсе (Excel → `prices.json`);
3. **если все курсы нашлись** — заполняет шаблон ответа ценой (или суммой при
   нескольких курсах) и отправляет письмо с платёжкой во вложении;
4. **если курс распознан не полностью** — перекладывает письмо в папку
   `HumanCheck` в ящике и не отвечает (разбирается вручную);
5. в обоих случаях пишет заявку в базу Supabase.

Всё работает как постоянная служба systemd, проверка почты — раз в 2 минуты.

## 2. Ключевые архитектурные решения

| Решение | Почему так |
|---|---|
| **Слоистая архитектура + внедрение зависимостей** | Каждый модуль отвечает за одно, зависимости передаются в конструктор → легко тестировать и менять. |
| **Ноль внешних зависимостей** | Только стандартная библиотека Node.js (≥20). Нечему протухнуть, нет уязвимостей в чужих пакетах, `npm install` не нужен. |
| **High-water-mark вместо «только непрочитанные»** | Бот помнит время последнего обработанного письма и берёт всё новее. Чинит баг «письмо открыли раньше бота». |
| **Идемпотентность (дедуп по id)** | Одно письмо не обрабатывается дважды, даже при перезапусках. |
| **Ретраи с экспоненциальным backoff** | Сетевой сбой или троттлинг Graph (429) не роняет бота — повтор с задержкой. |
| **Dead-letter** | Если база недоступна после всех повторов — заявка падает в `deadletter.jsonl`, не теряется. |
| **Graceful shutdown (SIGTERM)** | При перезапуске службы бот дописывает состояние и выходит чисто. |
| **Секреты и бизнес-данные не в git** | Токены, ключи, прайс и платёжка лежат только на сервере. |

## 4. Что в каждом файле

### Точка входа и запуск

**`src/index.js`** — *composition root*. Собирает граф зависимостей: создаёт
логгер, аутентификатор, клиент Graph, репозиторий базы, каталог цен, рендерер,
загружает вложение-платёжку, определяет адрес ящика и запускает поллер.
Здесь же — обработка сигналов `SIGTERM`/`SIGINT` для корректной остановки.

**`src/login.js`** — отдельная команда первичного входа (`npm run login`).
Показывает код и ссылку, ждёт, пока пользователь подтвердит вход на
microsoft.com, сохраняет токены. Запускается один раз.

### Инфраструктура

**`src/config.js`** — загружает настройки из `rules.json` и секреты из
`supabase.json`, валидирует их (**fail-fast**: при плохом конфиге падает сразу
с понятной ошибкой). Собирает единый объект конфигурации: пути к файлам,
параметры Graph, интервал опроса, кулдаун, имя папки HumanCheck и т.д.
Поддерживает переопределение через переменные окружения.

**`src/logger.js`** — структурный логгер с уровнями (`debug/info/warn/error`)
и контекстом. `child({msgId})` привязывает поля ко всем записям. Формат
`pretty` (человекочитаемый) или `json` (для машинной обработки).

**`src/util.js`** — мелкие утилиты без состояния:
- `sleep(ms)` — пауза;
- `readJson` — чтение JSON с fallback;
- `atomicWrite` — атомарная запись файла (через временный + rename), чтобы при
  падении не остаться с наполовину записанным состоянием;
- `withRetry(fn, opts)` — повтор с экспоненциальным backoff и джиттером,
  уважает `Retry-After`.

### Доступ к Microsoft и почте

**`src/auth.js`** — класс `Authenticator`. Хранит токены, автоматически
обновляет `access_token` по `refresh_token` (с защитой от параллельных
обновлений — *single-flight*). Умеет device code flow для первого входа.
Бросает `ReauthRequiredError`, если refresh-токен отозван (нужен повторный вход).

**`src/graph.js`** — два класса:
- `GraphClient` — HTTP-клиент к Microsoft Graph: подставляет токен, повторяет
  сетевые сбои/429/5xx через `withRetry`, уважает `Retry-After`.
- `MailService` — доменные операции с почтой:
  - `whoAmI()` — адрес ящика;
  - `fetchSince(iso)` — все письма новее отметки времени (с постраничностью);
  - `getBody(id)` — полный текст письма;
  - `reply(id, text)` — простой ответ;
  - `replyWithAttachment(id, text, attachment)` — ответ с вложением
    (createReply → задать тело → прикрепить файл → отправить);
  - `ensureFolder(name)` / `moveToFolder(id, folderId)` — папка HumanCheck.

### Состояние и хранение

**`src/state.js`** — класс `StateStore`, персистентное состояние поллера:
- `highWater` — ISO-время последнего обработанного письма;
- `processed` — множество id обработанных писем (идемпотентность, с лимитом);
- `lastReplyTo` — когда кому в последний раз отвечали (антиспам-кулдаун).
Запись атомарна.

**`src/supabase.js`** — класс `PaymentRepo`. Пишет заявки в таблицу
`payment_requests` через REST API Supabase, с ретраями. Если база недоступна
после всех повторов — заявка уходит в `deadletter.jsonl` (не теряется).

### Доменная логика

**`parser.js`** — чистый разбор текста заявки (без зависимостей, переиспользуется
в тестах). Формат письма — построчный «Метка: значение». Вытаскивает: курс,
дату проведения, организацию, ИНН, КПП, адрес, список ФИО. Устойчив к
косметике: звёздочки, прочерки-заполнители, несколько ФИО через запятую/«и».
Функция `looksLikePaymentRequest` определяет, заявка ли это вообще (по словам
«реквизиты», «счёт на оплату» или наличию ИНН).

**`src/catalog.js`** — класс `Catalog`, поиск курса(ов) в прайсе:
- `lookup(name)` — один курс, точное (после нормализации) совпадение;
- `match(text)` — **несколько курсов** из одной строки (через «и»/запятую).
  Хитрость: ищет в тексте **целые известные названия из прайса** (не режет по
  «и»/запятой, потому что названия сами их содержат). Возвращает
  `{ items, complete }`: `complete=false`, если остался нераспознанный кусок.
Нормализация: регистр, лишние пробелы, кавычки, ё→е, пунктуация.

**`src/render.js`** — класс `ReplyRenderer`. Подставляет в шаблон
(`reply-template.txt`) итоговую стоимость и построчную разбивку по курсам.
Один курс — вывод как в исходном образце; несколько — сумма + разбивка.
`formatPrice(8000)` → `"8 000"`.

**`src/classify.js`** — `matchRule(rules, msg, text)`: сопоставление письма с
правилами из конфига (по отправителю/теме/ключевым словам). Задел на будущее,
сейчас правил нет.

### Главная логика и цикл

**`src/processor.js`** — класс `MessageProcessor`, обработка **одного** письма
(use-case). Порядок решений:
1. уже обработано (дубль) → пропуск;
2. от себя / пусто → пропуск (защита от петли ответов);
3. рассылка/no-reply → пропуск;
4. кулдаун (недавно отвечали этому отправителю) → пропуск;
5. читает полный текст, проверяет `looksLikePaymentRequest`:
   - **заявка** → `catalog.match(курс)`:
     - все курсы распознаны → запись в базу + ответ шаблоном с ценой + платёжка;
     - не все → запись в базу (статус `needs_review`) + перенос в HumanCheck;
   - **не заявка** → проверка правил; если правило есть — ответ, иначе ничего.

**`src/poller.js`** — класс `Poller`, бесконечный цикл:
«забрать письма новее highWater → обработать каждое → продвинуть highWater →
сохранить состояние → записать health → пауза». При первом запуске стартует с
текущего момента (не отвечает на весь архив). Поддерживает мягкую остановку.

## 5. Поток обработки заявки (пошагово)

```
Письмо приходит в ящик
        │
        ▼
[poller] fetchSince(highWater) — забирает все новые письма
        │
        ▼
[processor] для каждого письма:
        │
        ├─ дубль / от себя / рассылка / кулдаун ──► пропуск
        │
        ▼
   looksLikePaymentRequest(текст)?
        │
        ├─ нет ──► правила (сейчас пусто) ──► ничего
        │
        ▼ да
   parser.parsePaymentRequest(текст) → { курс, ИНН, ФИО, ... }
        │
        ▼
   catalog.match(курс) → { items, complete }
        │
        ├─ complete=false ──► [supabase] запись (needs_review)
        │                     [graph] перенос письма в папку HumanCheck
        │                     (ответа НЕТ — разбор вручную)
        │
        ▼ complete=true
   [supabase] запись заявки (matched)
   [render] заполнить шаблон: сумма цен + разбивка по курсам
   [graph] replyWithAttachment: ответ + платёжка во вложении
        │
        ▼
   [state] отметить письмо обработанным, продвинуть highWater, сохранить
```

## 6. Аутентификация в Microsoft

Используется публичный client ID **«Microsoft Graph Command Line Tools»** —
это официальное приложение Microsoft. Своё приложение в Azure регистрировать
не нужно (личным аккаунтам портал Azure недоступен). Вход — через **device
code flow**: пользователь один раз вводит код на microsoft.com, дальше бот
работает по сохранённым токенам, обновляя их автоматически.

> ⚠️ На **корпоративной** почте (Microsoft 365) такой вход виден админам и
> часто заблокирован политиками безопасности. Подходит для личного ящика.

## 7. База данных (Supabase)

Таблица `payment_requests` (см. `schema.sql`):

| Колонка | Что хранит |
|---|---|
| `id`, `created_at` | служебные |
| `received_at` | когда пришло письмо |
| `from_email`, `subject` | отправитель и тема |
| `course` | курс(ы) — при нескольких через ` + ` |
| `event_date` | дата проведения (для вебинаров) |
| `org_name` | организация-плательщик |
| `inn`, `kpp` | ИНН / КПП |
| `postal_address` | почтовый адрес |
| `students` | список ФИО слушателей (jsonb) |
| `raw_body` | полный текст письма (на случай ошибок разбора) |
| `status` | `matched` (ответили) / `needs_review` (в HumanCheck) |

Доступ закрыт RLS: пишет только бот со `service_role`-ключом.

## 8. Запуск и эксплуатация

```bash
# первичный вход в почту (один раз)
npm run login

# постоянная работа
npm start

# проверка без отправки (логирует, что сделал бы)
npm run dry-run

# тесты
npm test
```

Как служба systemd (`mail-autoreply-bot`, `Restart=always`):

```bash
systemctl status mail-autoreply-bot      # статус
journalctl -u mail-autoreply-bot -f      # логи в реальном времени
systemctl restart mail-autoreply-bot     # перезапуск
```

**Переменные окружения** (необязательные): `POLL_SECONDS`,
`REPLY_COOLDOWN_HOURS`, `LOG_LEVEL`, `LOG_FORMAT` (pretty|json), `DRY_RUN=1`.

## 9. Как обновить данные

- **Прайс / платёжка**: заменить `prices.json` и `payment-order.xls` на сервере
  (или прислать новые файлы на почту — заберутся из вложений). Бот подхватит
  при следующем перезапуске.
- **Шаблон ответа**: отредактировать `reply-template.txt` (плейсхолдеры
  `{{total}}` — сумма, `{{breakdown}}` — построчная разбивка по курсам).
- **Схема базы**: `schema.sql` в Supabase SQL Editor.

## 10. Надёжность — что защищает от сбоев

- **Ретраи** на сетевые сбои и троттлинг Graph;
- **Dead-letter** при недоступности базы — ничего не теряется;
- **Идемпотентность** — нет двойных ответов;
- **Кулдаун** — одному отправителю не чаще раза в сутки;
- **Пропуск рассылок/no-reply** и **писем от себя** (защита от петли);
- **HumanCheck** — при неуверенности бот молчит, а не шлёт неверную цену;
- **health.json** — время и итог последнего цикла для мониторинга;
- **Graceful shutdown** — состояние сохраняется при остановке.
